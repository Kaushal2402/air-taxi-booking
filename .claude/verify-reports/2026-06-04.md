## 🔴 Critical Bugs

### BUG-001
- **File:** `admin-panel/src/App.tsx` + `admin-panel/src/components/layout/NavRail.tsx`
- **Module:** 03 — Roles & Access (RBAC)
- **Issue:** Module 03 is entirely unbuilt. The `NavRail` has a nav item `{ id: 'rbac', label: 'Roles & Access', path: '/rbac' }` but no pages exist and no route is registered in `App.tsx`. Clicking the sidebar item triggers the catch-all `<Route path="*" element={<Navigate to="/login" replace />} />`, which bounces the authenticated user to `/login` and then back to `/dashboard`.
- **Impact:** The entire RBAC module is inaccessible. Missing screens: `RolesListScreen`, `RoleEditorScreen`, `PermissionsCatalogScreen`.
- **Fix:** Build the RBAC module pages under `admin-panel/src/pages/rbac/`, register routes in `App.tsx`, and create backend endpoints for roles and permissions.

### BUG-004
- **File:** `backend/app/api/v1/router.py` (missing line) + `backend/app/api/v1/endpoints/operators.py:38`
- **Module:** 10 — Aircraft & Crew (Pilots sub-module)
- **Issue:** `pilots_router = APIRouter()` is defined in `operators.py` with 5 endpoints (`GET /`, `POST /`, `GET /{id}`, `PATCH /{id}`, `POST /{id}/approve`), but it is **never registered** in `router.py`. Every call from `PilotsCrewPage.tsx` to `/api/v1/pilots/*` returns HTTP 404.
- **Impact:** The entire Pilots & Crew management UI (`PilotsCrewPage.tsx`) is broken. Users cannot list, create, update, or approve pilot records. The page renders but all API calls silently fail.
- **Fix:** Add to `backend/app/api/v1/router.py`:
  ```python
  from app.api.v1.endpoints.operators import pilots_router
  api_router.include_router(pilots_router, prefix="/pilots", tags=["Pilots"])
  ```

## Backend Deep-Audit Notes

### Auto-commit pattern (correct)
`backend/app/database.py:get_db()` yields the session and calls `await session.commit()` on success and `await session.rollback()` on exception. Services that use `db.flush()` (for getting server-generated IDs before response serialization) are correct — the final commit happens at request teardown.

### Explicit commit pattern (correct)
Services that call `await db.commit()` explicitly (e.g., `driver_service.py`, `vehicle_service.py`, `operator_service.py`) all call `await db.refresh(obj)` immediately after before returning the ORM object. The subsequent auto-commit from `get_db()` is a no-op.

### Delete operations (correct)
`await db.delete(obj)` + `await db.commit()` with no `refresh()` — correct, since the object is deleted and the response is a MessageResponse or None.

### KYC endpoint (correct)
`kyc.py` calls `db.commit()` and `db.refresh()` directly inside the endpoint (not in a service). This is fine because the auto-commit from `get_db()` is a harmless no-op after an already-committed session.
